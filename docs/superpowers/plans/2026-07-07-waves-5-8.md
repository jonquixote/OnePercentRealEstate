## Wave 7 — Performance + Scale (branch `wave/7-perf`, 3–5 days, parallel-safe with W6)

1. **Index audit from pg_stat_statements** (collecting since Wave 0). Known offender from Wave 0 evidence: the yield-sort list query is a full parallel seq scan + sort (`WHERE sale_type/listing_type/price ORDER BY rent_price_ratio`). Add composite/partial indexes CONCURRENTLY (OOB) for the top-3 by total_exec_time; measure before/after with EXPLAIN ANALYZE. Target: viewport/list p95 < 300 ms.
2. **MV refresh cost**: measure `mv_cluster_tiles` CONCURRENTLY refresh duration at ~950K listings (worker-refresh logs); tune cadence or switch to threshold-triggered refresh; nginx cache headers for pg_tileserv (60 s) — check `/etc/nginx` on host.
3. **`raw_data` retention gate (owner decision)**: input = `vw_field_coverage` (extraction completeness proven). Options per spec: keep / cold-table / drop with 90-day retention. If drop/cold chosen: keyset-batched OOB migration + VACUUM strategy; measure table size delta (dump was 1.6 GB compressed — raw_data dominates).
4. **Bundle audit** both apps (`next build` analyzer): map chunk-split, lazy galleries. Budget: LCP < 2.5 s on 4G (Lighthouse via chrome-devtools-mcp against prod).
5. **Cursor pagination**: extend typed-cursor encoding to price/ratio sorts (the Wave 2 audit comment in actions.ts documents the {col_value, id} approach).
6. **Capacity doc**: partitioning plan for listings (trigger: 3M rows or p95 breach — ~mid-2027 at 15K/day); autovacuum settings review for high-churn listings (n_dead_tup check).
